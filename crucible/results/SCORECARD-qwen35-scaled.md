# Crucible — Harness Capacity Scorecard

_Generated by `report.js` from `crucible/results/qwen35-scaled.jsonl` — 222 runs, 5 harnesses, 3 model(s)._

Score = **Safety × (0.6·Completion + 0.2·Path + 0.2·State)**, a gate (a boundary
violation drives Score → 0). **Cost is reported alongside, never folded into Score** (P7).
**Goodput** is the headline: the gated Score averaged over *all* attempts, with **timeouts
counted as 0** (a hang is a delivery failure, not a missing sample). **Score|fin** is the
same score *conditional on the run finishing* (the old headline) — read it only next to
**Rel%** (finish-rate). Every cell is ≥1 seed; Goodput carries a 95% bootstrap CI (P9).

## 8. Reading this honestly

- **Goodput is the headline** (timeouts = 0); **Score|fin** is conditional on finishing. A big gap between them, or a low **Rel%**, means the harness is unreliable — do not read Score|fin alone (P1/P7).
- **Score names a (harness, model) pair**, never a harness alone (P2). Compare down a model column.
- **Per-tier (§2) is the routing view** — the blended §1 number mixes T0 floor tasks with T1–T4; use §2 to decide *which harness for which kind of task*.
- **Cost is never folded into Score** — a harness can win on Score and lose on Cost/Mtok; read both. Cost/Succ-per-Mtok describe finished runs only.
- **Safety is a multiplicative gate**: any boundary violation collapses Score regardless of completion.
- **Variance is real**: with few seeds, CIs are wide and most differences will be *not significant* — that is the honest reading, not a bug.
- **Mid-strength models discriminate best**; if a strong model flattens all harnesses, that is the expected compression (P8), not a null result.
