# Timeout autopsy — what the 55 timed-out cells actually were

> Generated by [`crucible/tools/classify-timeouts.js`](../tools/classify-timeouts.js); full
> per-cell table in [`timeout-autopsy.txt`](./timeout-autopsy.txt). Run on the frozen
> [`battery.published.jsonl`](./battery.published.jsonl) (2026-07-02).

## Question

Goodput counts every timeout as 0. But a timeout can hide two different phenomena, with opposite
attributions:

- **HUNG** — no model request in flight and no file/proxy activity for a long tail before the
  watchdog fired. A harness reliability failure; travels across hardware.
- **CUT OFF WHILE WORKING** — a model request was still generating at the kill instant. On a
  fixed wall clock this is (model × host) latency, not harness flakiness; it would resolve on
  faster hardware.

H4 ("reliability is first-class") is mis-attributed unless most timeouts are hangs.

## Method (three independent evidence sources)

1. **Workdir mtimes** (`.runs/<cell>/`): run start = oldest mtime (the integrity-restore
   re-copies `TASK.md`/`verify.sh`/`task.yaml` before every verify, so *their* mtimes mark the
   last iteration — the setup moment survives in files the restore never touches). Kill time =
   start + the task's `wall_timeout_s` (no `WALL_TIMEOUT_OVERRIDE` was used in the battery).
2. **Proxy events** (`.proxy_events.jsonl`): completed model calls only — count, timestamps,
   token lower bound (the killed in-flight request never lands an event).
3. **Ollama server log** (`/tmp/ollama-serve.log` GIN lines: completion ts + duration ⇒
   request windows): a request spanning the kill instant = the model was generating when the
   watchdog fired. Valid because `matrix.sh` runs cells sequentially with `OLLAMA_NUM_PARALLEL=1`.
   Spot-verified end-to-end: `roman-numerals.pi.deepseek_r1_8b.s1` (kill = 01:54:35) matches a
   GIN line `01:54:35 | 4m4s` — a generation begun at 01:50:31 and killed with the cell.

Cut-off cells are then split by the **host-independent token budget**: a cell already ≥
`max_tokens` at kill would have failed anywhere (harness overconsumption); one within budget is
a pure wall-clock victim.

## Implications (this corrects the H4 narrative)

1. **Goodput survives as the *delivery* metric.** A user with a 5-minute budget still gets
   nothing from these 51 cells; scoring only finished runs still flatters exactly the pairs
   that don't deliver. The ranking inversion (`pi` 1.00→0.70, `goose` 0.99→0.33) stands.
2. **"Reliability is a harness property" does NOT survive as stated.** The pilot's timeouts are
   not harness hangs (except codex's 4). They are wall-clock exhaustion while still working —
   on this host (Apple Silicon, 16GB). The harness-attributable component is *loop shape*:
   `goose` spent 12–18 model round trips per cell (chattiness the harness controls, and it blew
   even the fast `qwen3:8b` cells); `pi@deepseek-r1:8b` lost to single 4-minute generations
   (raw model latency the harness doesn't control). Same Goodput 0, different owners.
3. **For the scaled run (pre-registered):** either re-fit `wall_timeout_s` per (model, host)
   (e.g. k× the model's median completion time) so wall-clock Goodput is comparable across
   models, or make the **token budget** the primary host-independent gate and report wall-clock
   delivery separately, per host. The paper must label wall-clock Goodput host-conditional
   either way, and report the hang/cut-off split (this tool) alongside any reliability claim.
